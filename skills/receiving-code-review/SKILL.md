# 接收审查反馈

## 概述

代码审查需要技术评估，而非情绪化表演。

**核心原则：** 实现前验证。实现前先问。技术正确性优先于社交舒适。

## 响应模式

```
当收到审查反馈时：

1. 读取：完整阅读反馈，不要反应
2. 理解：用自己的话重述需求（或询问）
3. 验证：对照代码库实际情况检查
4. 评估：对这个代码库来说技术上成立吗？
5. 响应：技术确认或有理由的回推
6. 实现：逐项实现，每项测试
```

## 禁止的响应

**绝对不要：**
- "你说得对！"（表演性同意）
- "好观点！" / "优秀反馈！"（表演性）
- "让我立刻实现"（验证前）
- 任何感激表达（"谢谢指出"）

**替代做法：**
- 重述技术需求
- 问澄清问题
- 用技术推理回推如果不对
- 直接开始工作（行动 > 言辞）

## 处理不清晰的反馈

```
如果任何项不清晰：
  停止 — 不要实现任何东西
  询问不清晰项的澄清

为什么：项之间可能相关。部分理解 = 错误实现。
```

## 按来源处理

### 来自你的主管（your human partner）
- **信任** — 理解后实现
- **范围不清晰时仍要问**
- **无表演性同意**
- **跳到行动**或技术确认

### 来自外部审查者

```
实现前：
  1. 检查：对这个代码库技术上是正确的吗？
  2. 检查：会破坏已有功能吗？
  3. 检查：当前实现的理由是什么？
  4. 检查：在所有平台/版本上工作吗？
  5. 检查：审查者理解完整上下文吗？

如果建议看起来有问题：
  用技术推理回推

如果不能轻易验证：
  说："没有[X]我无法验证这个。我应该[调查/询问/继续]吗？"

如果与主管的先前决策冲突：
  停止，先与主管讨论
```

## YAGNI检查

```
如果审查者建议"正确实现"：
  搜索代码库中实际使用

  如果未使用："这个端点没有被调用。移除它（YAGNI）？"
  如果被使用：然后正确实现
```

## 实现顺序

```
对于多项反馈：
  1. 首先澄清任何不清晰的内容
  2. 然后按此顺序实现：
     - 阻塞问题（破坏性、安全性）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 每项修复单独测试
  4. 验证无回归
```

## 何时回推

在以下情况回推：
- 建议破坏已有功能
- 审查者缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 在这个技术栈上技术上不正确
- 遗留/兼容性原因存在
- 与主管的架构决策冲突

**如何回推：**
- 使用技术推理，而非防御心理
- 问具体问题
- 引用有效的测试/代码
- 如果涉及架构决定，让主管参与

## 纠正自己的回推

如果你回推了但是错了：
```
✅ "你对了——我检查了[X]它确实做了[Y]。正在实现。"
✅ "验证了，你是对的。我最初的理解是错的因为[原因]。修复中。"

❌ 长道歉
❌ 为回推辩护
❌ 过度解释
```

陈述纠正，事实性的，继续。

## 常见错误

| 错误 | 纠正 |
|------|------|
| 表演性同意 | 陈述需求或直接行动 |
| 盲目实现 | 先对照代码库验证 |
| 不测试就批量修改 | 一次一个，每项测试 |
| 假设审查者是对的 | 检查是否破坏东西 |
| 回避回推 | 技术正确性 > 舒适 |
| 部分实现 | 先澄清所有项 |
| 无法验证但继续 | 陈述限制，请求方向 |

## GitHub 讨论回复

在GitHub上回复内联审查评论时，在评论线程中回复（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而非作为顶级PR评论。

## 底线

**外部反馈 = 需要评估的建议，不是必须遵循的命令。**

验证。质疑。然后实现。

无表演性同意。始终技术严谨。
